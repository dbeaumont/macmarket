# Audit de la couche base de données — MacMarket

**Date d'audit** : 2026-07-07

**Périmètre analysé** : migrations Flyway V1 → V8 (9 scripts SQL), 12 entités JPA, 8 repositories Spring Data, 2 requêtes SQL natives (`nativeQuery = true`), configuration datasource (`application.yml`, `application-dev.yml`, `application-docker.yml`), ADR-0004

**Contexte** : Application e-commerce macOS simulée, monolithe modulaire Spring Modulith, architecture DDD hexagonale (ADR-0002), PostgreSQL 17 (ADR-0004), Flyway pour les migrations, `ddl-auto=none`.

---

## Résumé exécutif

Le schéma est globalement bien conçu : convention de nommage snake\_case cohérente, préfixes par bounded context, `ddl-auto=none` avec Flyway, types NUMERIC pour les montants, UUID comme clés primaires. Les bases sont saines.

Quatre problèmes méritent une attention immédiate. L'utilisation de `FetchType.EAGER` sur trois associations `@OneToMany` constitue un risque de performance sérieux qui se manifestera sous charge. La table `payment_payments` stocke un `order_id` sans contrainte de clé étrangère : l'intégrité référentielle n'est pas garantie au niveau base de données. Le module `admin` mappe directement ses entités JPA sur des tables appartenant aux bounded contexts `order` et `catalog`, ce qui crée un couplage structurel invisible à la compilation. Enfin, le pool de connexions HikariCP n'est pas configuré, ce qui peut entraîner une saturation silencieuse en production.

---

## 1. Conception du schéma

### 1.1 Nommage

#### Constats

La convention snake\_case est respectée partout. Les tables sont préfixées par bounded context conformément à ADR-0004 (`catalog_products`, `cart_carts`, `order_orders`, `payment_payments`, `admin_daily_stats`, `user_shipping_profiles`). Les noms d'index suivent le patron `idx_<table>_<colonne>`.

Les contraintes de clé étrangère sont anonymes dans V1, V2 et V4 :

```sql
-- V1 — contrainte anonyme
product_id UUID NOT NULL REFERENCES catalog_products(id) ON DELETE CASCADE,

-- V2 — contrainte anonyme
cart_id UUID NOT NULL REFERENCES cart_carts(id) ON DELETE CASCADE,

-- V4 — contrainte anonyme
order_id UUID NOT NULL REFERENCES order_orders(id) ON DELETE CASCADE,
```

Seule V8 utilise un nom explicite :

```sql
-- V8 — contrainte nommée
CONSTRAINT uk_user_shipping_profiles_user UNIQUE (user_id)
```

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MINEUR | `catalog_product_specs`, `cart_items`, `order_items` | Contraintes FK anonymes — rend le diagnostic d'erreur difficile (`violates foreign key constraint "xxx"` sans nom descriptif) | Nommer : `fk_catalog_product_specs_product`, `fk_cart_items_cart`, `fk_order_items_order` |

---

### 1.2 Typage et structure

#### Constats

Les montants utilisent `NUMERIC(10,2)` ✅. Les identifiants techniques sont des UUID avec `gen_random_uuid()` ✅. Les colonnes obligatoires portent `NOT NULL` ✅.

Deux colonnes génériques méritent d'être signalées.

`cart_carts.user_id` et `order_orders.user_id` sont typées `VARCHAR(255)`. Les sujets Keycloak sont des UUID au format `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` (36 caractères). Un `VARCHAR(255)` est surdimensionné sans être un problème bloquant.

`order_items.subtotal` est une colonne dérivée (= `unit_price × quantity`). Cette dénormalisation n'est pas documentée dans le DDL.

`event_publication.status` est un `VARCHAR(20)` libre sans contrainte `CHECK` sur les valeurs admises.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MINEUR | `cart_carts.user_id`, `order_orders.user_id` | `VARCHAR(255)` surdimensionné pour un UUID Keycloak (36 car.) | Utiliser `VARCHAR(36)` ou le type `UUID` natif PostgreSQL |
| MINEUR | `order_items.subtotal` | Colonne dérivée non documentée — dénormalisation silencieuse | Ajouter un commentaire SQL (`COMMENT ON COLUMN`) précisant l'intention : conserver la valeur calculée au moment de la commande |
| MINEUR | `event_publication.status` | `VARCHAR(20)` sans contrainte `CHECK` sur les valeurs | Ajouter `CHECK (status IN ('PUBLISHED', 'COMPLETED', 'FAILED'))` ou utiliser un type `ENUM` PostgreSQL |

---

### 1.3 Intégrité référentielle

#### Constats

La quasi-totalité des relations parent-enfant est protégée par des FK avec `ON DELETE CASCADE`, ce qui est cohérent avec la sémantique d'agrégat (supprimer un panier supprime ses lignes).

Exception notable : `payment_payments.order_id` référence implicitement `order_orders.id`, mais **aucune contrainte FK n'est déclarée** :

```sql
-- V5 — UNIQUE mais pas de FK déclarée
CREATE TABLE payment_payments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id        UUID NOT NULL UNIQUE,   -- ← pas de REFERENCES order_orders(id)
    ...
);
```

Il est possible d'insérer un paiement avec un `order_id` inexistant. La contrainte d'unicité protège contre les doublons mais pas contre les orphelins.

Les `product_id` dans `cart_items` et `order_items` ne sont volontairement pas contraints vers `catalog_products` — ce choix découple les bounded contexts et préserve le snapshot produit. Ce pattern est acceptable en DDD, mais devrait être documenté.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MAJEUR | `payment_payments.order_id` | Absence de contrainte FK vers `order_orders(id)` — un paiement orphelin peut exister en base | Ajouter : `CONSTRAINT fk_payment_order FOREIGN KEY (order_id) REFERENCES order_orders(id)` |
| MINEUR | `cart_items.product_id`, `order_items.product_id` | Couplage inter-bounded-context intentionnellement évité, mais non documenté | Ajouter un commentaire SQL explicitant le choix de snapshot |

---

### 1.4 Normalisation

#### Constats

Le schéma respecte globalement la 3NF. Les snapshots produit dans `cart_items` et `order_items` (`product_name`, `product_image`, `unit_price`) sont une dénormalisation intentionnelle et légitime pour capturer les données au moment de l'acte.

`order_items.subtotal` est une colonne calculée (valeur = `unit_price × quantity`). Elle n'est pas justifiée par des besoins de performance documentés et introduit un risque de désynchronisation si la valeur n'est pas recalculée lors d'une correction.

`admin_daily_stats` est une table d'agrégats précalculés. Ce pattern est acceptable pour les besoins de reporting, mais aucune logique de recalcul ou de cohérence n'est garantie au niveau base de données.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MINEUR | `order_items.subtotal` | Colonne dérivée non protégée par une contrainte `CHECK` — risque de désynchronisation | Ajouter `CONSTRAINT ck_order_items_subtotal CHECK (subtotal = unit_price * quantity)` pour garantir la cohérence |

---

### 1.5 Index et performance

#### Constats

La couverture d'index est généralement bonne. Deux index sont **redondants** car les contraintes `UNIQUE` créent automatiquement un index B-tree en PostgreSQL.

```sql
-- V1 — l'index est créé automatiquement par UNIQUE sur slug
slug VARCHAR(200) NOT NULL UNIQUE,
CREATE INDEX idx_catalog_products_slug ON catalog_products(slug);  -- ← redondant
```

```sql
-- V8 — l'index est créé automatiquement par la contrainte UNIQUE
CONSTRAINT uk_user_shipping_profiles_user UNIQUE (user_id),
CREATE INDEX idx_user_shipping_profiles_user ON user_shipping_profiles(user_id);  -- ← redondant
```

La requête native de recherche filtrée dans le catalogue combine `active`, `category`, `price`, `name ILIKE` :

```sql
WHERE (:active IS NULL OR p.active = :active)
  AND (:category IS NULL OR p.category = :category)
  AND (:minPrice IS NULL OR p.price >= :minPrice)
  ...
```

Le cas d'usage le plus fréquent (produits actifs d'une catégorie) bénéficierait d'un index composite `(active, category)`.

Pour `AdminOrderReadRepository`, les requêtes `countOrdersSince`, `sumRevenueSince` filtrent sur `created_at >= :since`. L'index `idx_order_orders_created` sur `created_at` couvre ce cas ✅.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MINEUR | `idx_catalog_products_slug` | Redondant avec la contrainte UNIQUE sur `slug` | Supprimer dans une migration `V9` |
| MINEUR | `idx_user_shipping_profiles_user` | Redondant avec la contrainte UNIQUE sur `user_id` | Supprimer dans une migration `V9` |
| MINEUR | `catalog_products` | Absence d'index composite `(active, category)` pour la recherche filtrée principale | Ajouter `CREATE INDEX idx_catalog_products_active_category ON catalog_products(active, category)` |

## 4. Accès cross-bounded-context

#### Constats

Le module `admin` mappe ses propres entités JPA directement sur les tables des bounded contexts `order` et `catalog` :

```java
// admin module — AdminOrderJpaEntity.java
@Entity
@Table(name = "order_orders")          // ← table appartenant au bounded context "order"
public class AdminOrderJpaEntity { ... }

// admin module — AdminOrderItemJpaEntity.java
@Entity
@Table(name = "order_items")           // ← table appartenant au bounded context "order"
public class AdminOrderItemJpaEntity { ... }

// admin module — AdminProductJpaEntity.java
@Entity
@Table(name = "catalog_products")      // ← table appartenant au bounded context "catalog"
public class AdminProductJpaEntity { ... }
```

Ce pattern CQRS côté lecture est fonctionnel mais crée un couplage structurel entre le module `admin` et les schémas des modules `order` et `catalog`. Toute modification de colonne dans `order_orders` ou `catalog_products` impacte silencieusement le module `admin`, sans erreur de compilation.

C'est une violation des principes d'isolation de bounded contexts de l'architecture DDD hexagonale (ADR-0002) : *"chaque bounded context est autonome et expose ses fonctionnalités via des ports explicites"*.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MAJEUR | `admin` → `order_orders`, `order_items`, `catalog_products` | Accès direct cross-bounded-context par mapping JPA partagé — violation ADR-0002 | Exposer des vues de lecture dédiées (`admin_v_orders`, `admin_v_products`) ou passer par des événements de domaine pour alimenter une projection `admin` isolée |

## 5. Sécurité

#### Constats

Aucune donnée sensible (mot de passe, token) n'est stockée dans la base applicative — l'authentification est entièrement déléguée à Keycloak ✅.

Les requêtes `@Query` utilisent systématiquement des paramètres nommés (`:param`), pas de concaténation de chaînes ✅. La recherche par `ILIKE` est paramétrée :

```java
// Correct — pas de risque d'injection SQL
.name ILIKE CONCAT('%', :search, '%')
```

Le mot de passe de la datasource est externalisé en variable d'environnement dans `application.yml` ✅.

Le profil dev (`application-dev.yml`) définit une valeur par défaut non vide pour le mot de passe :

```yaml
# application-dev.yml
password: ${DB_PASSWORD:macmarket_secret}
```

Ce mot de passe par défaut est committé dans le dépôt. Acceptable en développement local, mais à surveiller si ce profil venait à être activé en dehors du poste développeur.

Aucun fichier de configuration des rôles PostgreSQL n'est présent dans les migrations — les droits de l'utilisateur applicatif ne sont pas documentés ni contraints.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MINEUR | `application-dev.yml` | Mot de passe par défaut `macmarket_secret` committé dans le dépôt | Utiliser une valeur vide ou un `.env.local` non versionné comme fallback |
| MINEUR | Migrations Flyway | Aucune migration ne configure les droits minimaux de l'utilisateur applicatif | Ajouter une migration ou un script d'initialisation `GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO macmarket` |

## 6. Couche JPA et configuration

### 6.1 FetchType et risque N+1

Trois associations `@OneToMany` déclarent `FetchType.EAGER` :

```java
// OrderJpaEntity.java
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true,
           fetch = FetchType.EAGER)     // ← EAGER sur OneToMany
private List<OrderItemJpaEntity> items;

// CartJpaEntity.java
@OneToMany(mappedBy = "cart", cascade = CascadeType.ALL, orphanRemoval = true,
           fetch = FetchType.EAGER)     // ← EAGER sur OneToMany
private List<CartItemJpaEntity> items;

// AdminOrderJpaEntity.java
@OneToMany(mappedBy = "order", fetch = FetchType.EAGER)     // ← EAGER sur OneToMany
private List<AdminOrderItemJpaEntity> items;
```

En Hibernate, `EAGER` sur `@OneToMany` génère une requête `IN` supplémentaire pour chaque agrégat parent chargé, ou force un `JOIN FETCH` non contrôlé. Sur les pages de liste de commandes (`findAllByOrderByCreatedAtDesc(Pageable)`), chaque commande déclenchera une requête supplémentaire pour ses lignes. L'impact est directement proportionnel à la taille de la page.

La bonne pratique recommandée est `FetchType.LAZY` systématique sur `@OneToMany`, avec un `JOIN FETCH` explicite dans les requêtes qui ont besoin des données enfants.

### 6.2 Requêtes natives avec syntaxe PostgreSQL spécifique

Deux `nativeQuery = true` utilisent `ILIKE` (extension PostgreSQL) et `DATE()` :

```java
// ProductSpringDataRepository.java
@Query(value = "... p.name ILIKE CONCAT('%', :search, '%') ...", nativeQuery = true)

// AdminOrderReadRepository.java
@Query(value = "SELECT DATE(created_at) as day, ...", nativeQuery = true)
```

Ces constructions sont légitimes car ADR-0004 engage le projet sur PostgreSQL. Elles ne sont pas documentées comme telles.

### 6.3 Retour `List<Object[]>` non typé

Plusieurs méthodes de `AdminOrderReadRepository` retournent `List<Object[]>` :

```java
List<Object[]> countByStatus();
List<Object[]> revenueByDay(@Param("since") Instant since);
List<Object[]> customerAggregation(Pageable pageable);
List<Object[]> ordersByDay(@Param("since") Instant since);
```

Ce contrat est fragile : tout changement de projection dans la requête JPQL ou SQL casse silencieusement le code appelant. L'usage de Spring Data Projections (interfaces) ou de records DTO serait plus robuste.

### 6.4 Configuration datasource

`ddl-auto=none` ✅, Flyway activé ✅, `open-in-view: false` ✅.

Le dialecte Hibernate n'est pas explicitement configuré. Spring Boot détecte automatiquement `PostgreSQLDialect` via le driver JDBC, mais la configuration explicite permet d'éviter des surprises lors d'un changement de driver ou de version Spring Boot.

Le pool de connexions HikariCP utilise ses valeurs par défaut (`maximumPoolSize=10`, pas de `connectionTimeout`, pas de `maxLifetime` explicites). En production, ces valeurs sont à ajuster en fonction de la charge attendue.

`application-docker.yml` ne fournit pas de credentials : ils doivent être injectés via les variables d'environnement `DB_USERNAME` et `DB_PASSWORD` définies dans le `docker-compose.yml`. C'est correct, mais il n'y a pas de documentation sur les variables d'environnement requises.

#### Non-conformités

| Sévérité | Objet | Problème | Recommandation |
|---|---|---|---|
| MAJEUR | `OrderJpaEntity`, `CartJpaEntity`, `AdminOrderJpaEntity` | `FetchType.EAGER` sur `@OneToMany` — risque N+1 et chargement non contrôlé | Passer à `FetchType.LAZY` et utiliser `JOIN FETCH` dans les requêtes nécessitant les items |
| MAJEUR | HikariCP | Pool de connexions non configuré — saturation silencieuse possible en production | Configurer `spring.datasource.hikari.maximum-pool-size`, `connection-timeout`, `max-lifetime` dans `application.yml` |
| MINEUR | `AdminOrderReadRepository` | `List<Object[]>` comme type de retour — pas de type-safety | Introduire des interfaces de projection Spring Data ou des records DTO |
| MINEUR | `ProductSpringDataRepository`, `AdminOrderReadRepository` | Requêtes natives avec syntaxe PostgreSQL non documentée | Ajouter un commentaire `// PostgreSQL-only: ILIKE` et référencer ADR-0004 |
| MINEUR | `application.yml` | Dialecte Hibernate non configuré explicitement | Ajouter `spring.jpa.database-platform: org.hibernate.dialect.PostgreSQLDialect` |

## Recommandations prioritaires

1. **BDD-001** — Ajouter la contrainte FK manquante sur `payment_payments.order_id` dans une migration `V9`. Risque d'intégrité réelle en production.

2. **BDD-003** — Migrer les trois `@OneToMany` vers `FetchType.LAZY`. Introduire des requêtes avec `JOIN FETCH` explicite pour les cas d'usage nécessitant les items. C'est le changement le plus impactant sur les performances sous charge.

3. **BDD-004** — Configurer HikariCP avant tout déploiement en production (`maximum-pool-size`, `connection-timeout`, `idle-timeout`, `max-lifetime`).

4. **BDD-002** — Décider d'une stratégie d'isolation pour l'accès lecture du module `admin` : soit des vues PostgreSQL dédiées (`admin_v_orders`, `admin_v_products`) projetées à partir des tables sources, soit une projection event-driven alimentée par les domain events. L'approche vues est la moins intrusive à court terme.

5. **BDD-009 / BDD-010** — Supprimer les deux index redondants dans une migration `V9`.

6. **BDD-008** — Ajouter `CONSTRAINT ck_order_items_subtotal CHECK (subtotal = unit_price * quantity)` pour fiabiliser la colonne dérivée.
